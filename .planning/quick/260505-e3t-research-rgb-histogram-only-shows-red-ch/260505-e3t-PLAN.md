## HYPOTHESIS 1 — CIAreaHistogram output channel semantics (MOST LIKELY ROOT CAUSE)

CIAreaHistogram is documented to return a histogram image whose pixel values are
proportional to the bin counts. BUT: the output colorSpace is NOT guaranteed to be RGBA
with independent channels. The filter outputs to a linear display-referred colorspace. On
devices using a wide-gamut (P3) display pipeline, CIContext with no explicit workingColorSpace
will process the histogram output through the device's default display colorspace, which
is P3. The CIColorSpace used for the rendering path affects how CIContext.render maps the
values between the filter output and your Float buffer.

Key insight to document: CIAreaHistogram's output image is tagged with the WORKING
colorspace, not nil. When you call context.render(..., colorSpace: nil), nil means "match
the context's working colorspace" — it does NOT mean "raw bytes". Passing nil only bypasses
the destination→display transform, not the source→working transform. G and B channels
get an implicit per-channel matrix multiplication applied by the P3 → linear conversion
before landing in the float buffer.

Fix direction: Pass `CGColorSpaceCreateDeviceRGB()` (device-linear, no ICC curve applied)
as colorSpace, not nil. Alternatively, set a workingColorSpace on the CIContext itself:
`CIContext(options: [.workingColorSpace: CGColorSpaceCreateDeviceRGB()])`.

---

## HYPOTHESIS 2 — CIAreaHistogram scale=1.0 causes integer truncation

`area.scale` is documented as: "The scale factor applied to the histogram values." The
default value is 1.0, which means raw counts are returned. For a 3000x4000 image, a single
bin can receive millions of pixel votes. When these are read back as Float32, they are NOT
normalized to [0,1] — they are raw counts. The float buffer stores values like 8,432,100.0
in the R channel bin-0. However, when CIContext applies any color transform, it CLAMPS
floats to [0,1] before writing because it treats the output as color (not data).

Evidence for this: The red channel is the first channel. If ALL values are being clamped to
1.0 before writing, you'd see a flat solid fill, not a curve. But if only R survives as > 0
because the color transform matrix sends G and B through coefficients that happen to result in
0.0 for the actual image content... this is consistent with "only red visible".

Fix direction: Set `area.scale = 1.0 / Float(maxExpectedPixelCount)` to normalize inside
the filter before readback, eliminating any transform-induced clamp. MaxExpectedPixelCount
can be estimated as (imageWidth * imageHeight) / 256 (average pixels per bin). However the
exact scale depends on image size, so a safer approach is a two-pass: first read with scale=1
to discover the peak, then normalize in Swift (which the current code already does — but
if values are clamped to 1.0 before normalization, you lose the differential between channels).

## HYPOTHESIS 3 — GraphicsContext.blendMode on var copy does not take effect in Canvas

In HistogramOverlayView.swift, Pass 1 does:
```swift
var layer = ctx
layer.blendMode = .plusLighter
layer.fill(...)
```

The Swift docs for GraphicsContext state that it is a VALUE TYPE. Mutating a local `var layer`
copy DOES affect drawing operations performed through that copy — SwiftUI Canvas is designed
to support this pattern. So this is NOT a bug. However: the fill opacity is 0.30, and the
channel colors are `Color(red: 1.0, green: 0.22, blue: 0.22)` for red, `Color(red: 0.30,
green: 1.0, blue: 0.40)` for green, and `Color(red: 0.40, green: 0.60, blue: 1.0)` for blue.
With .plusLighter blending at 0.30 opacity on a black background:
- Red channel draws correctly (high red component)
- Green channel at 0.30 opacity = r:0.09, g:0.30, b:0.12 — VERY faint on black
- Blue channel at 0.30 opacity = r:0.12, g:0.18, b:0.30 — VERY faint on black

The strokes in Pass 2 use `channel.color.opacity(0.95)` which should be visible. BUT: if the
HistogramData itself has g[] and b[] all zeros (upstream bug), neither pass will show anything.
This hypothesis requires confirming the data is correct at the HistogramData level.

This is NOT the root cause if g[] and b[] arrays contain nonzero values. It IS a contributing
factor to readability even after fixing the data.

## HYPOTHESIS 4 — Input CGImage is already grayscale/single-channel

If `previewImage.cgImage` is backed by a kCVPixelFormatType_420YpCbCr8BiPlanarVideoRange or
a grayscale CGColorSpace, CIImage(cgImage:) will construct a CIImage that only has meaningful
luma data. When CIAreaHistogram processes this, all color information is absent — R, G, B
bins would all reflect the same luma distribution, which after normalization become identical
curves. This would NOT produce "only red visible" — it would produce three overlapping
identical curves. So this hypothesis explains something different from the symptom; downgrade
its likelihood. However: if the CGImage came through UIImage and through a UIGraphicsImageRenderer
round-trip, it could have been promoted to RGBA32 but with G=B=0 in the data (malformed).
Worth a quick check.

Verification: In recomputeHistogramIfVisible, add a debug log:
```swift
print("CGImage colorspace: \(cg.colorSpace?.name ?? "nil"), bpc: \(cg.bitsPerComponent), bpp: \(cg.bitsPerPixel)")
```

## HYPOTHESIS 5 — CIAreaHistogram output extent offset

CIAreaHistogram outputs a 256x1 image. However, in CIKit, filter output images frequently
have a non-zero origin. The code uses `CGRect(x: 0, y: 0, width: 256, height: 1)` as the
render bounds. If the actual output extent has y != 0 (e.g., y = -1 or y = some negative
float), passing bounds (0,0,256,1) to CIContext.render reads from the wrong scanline and
may return all zeros or garbage for non-red channels.

Fix direction: Use `bins.extent` as the bounds, not a hardcoded CGRect:
```swift
let bounds = bins.extent  // not CGRect(x:0, y:0, width:256, height:1)
```

This is a low-effort, high-confidence fix that should be applied regardless of whether it is
the primary root cause.
